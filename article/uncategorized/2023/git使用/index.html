Git 菜单

1 基础概念

  • 创建仓库

    • 空仓库
      • git init 将当前的目录转换成一个 Git 仓库
      • git init <directory> 在指定目录创建一个空的 Git 仓库
      • git init --bare <directory> 初始化一个裸的 Git 仓库，但是忽略工作目录。
    • 从远端拷贝仓库
      • git clone <repo>
      • git clone <repo> <directory>

  • 配置

  • 工作区、缓存区、本地仓库

    • 工作区内文件默认处于未追踪状态
      • git add < filename >可将工作区文件添加进缓存区，处于跟踪状态
      • git rm < filename > 将某个跟踪状态的文件设为未跟踪状态，同时文件也会被删除
      • git rm --cached < filename > 将某个跟踪状态的文件设为未跟踪状态，文件不会被删除
    • 跟踪状态
      • git add < filename > or git add *将工作区文件添加进缓存区
      • git commit -m "代码提交信息" 将缓存区内的修改提交至本地仓库
      • git checkout <existing-branch> 查看特定分支
    • 忽略文件
      • .gitignore文件

  • commit branch head tag 个人理解

    • commit是一次提交，本质是一次修改
    • branch是一个指针，用于指向一个commit,当head指向branch时，commit命令会创建一个commit，并移动branch
    • head指向当前用户可操作对象,git checkout commit-name会使head指向一个commit而不是branch，进入分离head状态，此时只可以查看commit的文件，无法
    • tag多用于表示项目里程碑式的节点（版本号）

2 常用操作

2.1 检查仓库状态

  • git status 列出已缓存、未缓存、未追踪的文件。
  • git log 显示已提交的快照
    • git log --oneline 将每个提交压缩到一行。
  • git diff 比较文件在暂存区和工作区的差异
  • git reflog 显示本地仓库的引用日志。
    • 如果不小心回滚或重写了项目历史，使用reflog回到未来
    • 每次当前的 HEAD 更新时（如切换分支、拉取新更改、重写历史或只是添加新的提交），引用日志都会添加一个新条目。

2.2 回滚错误的修改

  • git revert <commit> 生成一个撤销该commit的commit，然后将当前分支移动至该commit
    • 不会删除原有的提交记录，保留了原来的更改，用一个新的提交来实现撤销。
    • 相较于reset是最安全的方式
  • git reset <commit> 将当前分支的末端移到 <commit>，将缓存区重设到这个提交，但不改变工作目录。
    • 会删除原有的提交记录，但不会改变工作目录，允许用户用更干净、原子性的快照重新提交项目历史。
  • git reset --hard <commit> 将当前分支的末端移到<commit>，将缓存区和工作目录都重设到这个提交。
    • 默认属性为--mixed
    • 会同时修改提交记录和工作目录

2.3.1 分支

  • git branch
    • 列出仓库中所有分支。
  • git branch <branch-name>
    • 创建一个名为 <branch-name> 的分支。不会 自动切换到那个分支去。
    • git checkout -b <new-branch> 创建并查看 <new-branch>
    • git checkout -b <new-branch> <existing-branch> 将 <existing-branch> 作为新分支的基，而不是当前分支。
  • git branch -d <branch>
    • 删除指定分支。这是一个安全的操作，Git 会阻止你删除包含未合并更改的分支。
  • git branch -m <branch>
    • 将当前分支命名为 <branch>。

2.3.2 合并

  • git merge <branch>

    • 将指定分支并入当前分支,Git 自动决定使用哪种合并算法
    • 若当前分支是branch祖先，则进行快速向前合并，将当前分支指针移向branch
    • 若当前分支与branch分叉，则会新建一个commit，将当前分支指针移向commit，称为三路合并

  • git merge --no-ff <branch>

    • 将指定分支并入当前分支，但 总是 生成一个合并提交（即使是快速向前合并）

2.4 阶段性开发完成后重写项目历史

  • git commit --amend
    • 将缓存的修改和之前的提交合并到一起，而不是提交一个全新的快照。
    • 如果仓促commit后发现有个文件不正确，修改后可以使用--amend避免有多次提交记录。
    • --no-edit 标记会修复提交但不修改提交信息
  • git rebase <base> 将当前分支以及之前的提交都复制到base上。
    • base可以是任何类型的提交引用(ID、branch、tag，或是 HEAD 的相对引用)。
    • 将我的更改建立在其他人的进展之上
    • 相较于merge，rebase产生的是一个快速向前的合并以及完美的线性历史。
  • git rebase -i <base>
    • 开始交互式 rebase。交互式 rebase 给你在过程中修改单个提交的机会，而不是盲目地将所有提交都移到新的基上。你可以移除、分割提交，更改提交的顺序。
    • 它就像是打了鸡血的 git commit --amend 一样。
  • git cherry-pick commit1 commit2...

3 开发tricks

3.1 单人开发

  • 开发新功能
    • 可以新建一个分支new-feature，在new-feature上面每完成一小部分就提交一次，周而复始，完成新功能开发后切换回master分支，将new-featuremerge即可。
  • 合并
    • 单用merge
    • rebase+merge
